We need more tests

  • In order to move specs forward.
  • To avoid fragmentation / interop issues

To get more tests, we need more contributors

  • We should continue to crowd source testing via events like TestTWF and expand to add smaller, more focussed events.
  • The goal of these events should be not just to develop tests at a singe event but to build a new community of contributors that continue to contribute tests and throw their own events.
  • We can also get more contributors by hiring contractors.

New contributors will need to be supported in order to be successful

Current Situation:

  • It is hard for individual developers to get involved in the process:

  • Documentation is not consolidated and may be stale, is likely different per WG and in some cases may not even exist.

  • It takes new people (individuals, contractors or otherwise) some time to get set up:

  • Setting up HG is non trivial for some and for many it is different than the tools they are used to using, so it may take some time to learn.

  • It is sometimes unclear who to contact for access

  • Specs are hard to read / digest for newcomers and in order to make worthwhile tests, you need to understand the spec. No W3C docs on how to read a spec.

  • Not all specs use normative language.

  • It is unclear on where & how an individual can get started

  • Not (m)any sample test

  • No clear indication of what tests have been written or need to be written for a particular spec.

  • There isn't much of a support structure for new people.

  • Once tests are submitted, the submitter has not idea when they will be reviewed & approved.

  • Unreviewed/Approved test contributions do not necessarily help.

  • Individuals are not motivated to contribute more because there is no impact, reward or credit.

What would help:

  • - Infrastructure and documentation that will make the barrier to contributing very low.

  • There needs to be a clear place or places to get support and we need responders.

  • More normative language in specs.

  • Somehow list out tests that ned to be written for a given spec.

  • Set expectations on turnaround time of test reviews and minimize that time as much as possible.

  • Train more reviewers and provide a clear paths & requirements on how to become a reviewer.

  • Provide some sort of carrot for new contributors. Something that gives them a stake in test contribution and a reward for contributing.
